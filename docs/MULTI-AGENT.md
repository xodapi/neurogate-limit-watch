# MULTI-AGENT.md — Координация нескольких агентов

> Дополнение к AGENTS.md для случаев когда 2+ агента работают параллельно.
> Читать вместе с AGENTS.md.

---

## Проблема параллельной работы

Когда два агента берут разные Issue одновременно — они могут:
- Редактировать один и тот же файл → merge conflict
- Зависеть друг от друга → deadlock
- Оба добавить зависимость в Cargo.toml → конфликт lock-файла

Решение — **файловые зоны** и **явные зависимости между Issue**.

---

## 1. Файловые зоны (кто что трогает)

Каждый Issue получает label зоны. Два агента не берут Issue из одной зоны одновременно.

| Label зоны | Файлы |
|------------|-------|
| `zone:api` | `src/api.rs`, `src/cli/update.rs` |
| `zone:tui` | `src/monitor.rs`, `src/main.rs` |
| `zone:notify` | `src/notify.rs` |
| `zone:gui` | `ui/*.slint`, `src/bin/vimit-gui.rs` |
| `zone:ci` | `.github/workflows/*.yml` |
| `zone:docs` | `README*.md`, `SUBMISSION*.md`, `ROADMAP.md` |
| `zone:cargo` | `Cargo.toml`, `Cargo.lock`, `build.rs` |

### Правило зон

Перед тем как взять Issue — проверить:
```bash
gh issue list --state open --label "zone:gui" --assignee "*"
```
Если кто-то уже работает в этой зоне — **не брать** Issue из той же зоны.
Взять Issue из другой зоны или подождать.

---

## 2. Зависимости между Issue

В теле Issue может быть секция:
```
## Зависит от
- #12 (должен быть merged перед началом)
- #15 (должен быть merged перед началом)
```

Агент **не начинает** Issue если его зависимости не закрыты.
Проверить:
```bash
gh issue view <NUMBER> | grep "Зависит от"
gh issue view <DEP_NUMBER> --json state | jq .state
```
Если зависимость открыта — оставить комментарий и не брать задачу:
```bash
gh issue comment <NUMBER> \
  --body "⏳ Ожидаю закрытия #<DEP>. Вернусь после merge."
```

---

## 3. Алгоритм выбора задачи для агента

```
1. Получить список свободных Issues:
   gh issue list --state open --label agent --assignee ""

2. Для каждого Issue (по приоритету):
   a. Проверить зависимости → все closed?
   b. Определить zone: label
   c. Проверить что zone свободна:
      gh issue list --label "zone:<X>" --assignee "*" --state open
   d. Если зона свободна И зависимости closed → взять

3. Назначить себя + оставить комментарий с именем агента
```

## 4. Опасные файлы — только один агент

Эти файлы **никогда не редактируются параллельно**:

| Файл | Почему опасен |
|------|--------------|
| `Cargo.toml` | Конфликт зависимостей + Cargo.lock |
| `Cargo.lock` | Автогенерируется, всегда конфликт |
| `src/main.rs` | Точка входа, много агентов → chaos |
| `ui/app.slint` | Монолитный UI файл |
| `.github/workflows/ci.yml` | CI логика |

Для этих файлов действует правило: **label `zone:cargo` / `zone:ci` / `zone:gui` — не более одного активного Issue одновременно**.

## 5. Rebase-стратегия для параллельных веток

Когда твой PR готов, но в main уже появились новые коммиты от другого агента:

```bash
git fetch origin main
git rebase origin/main

# Если конфликт в Cargo.lock — не решать вручную:
cargo generate-lockfile
git add Cargo.lock
git rebase --continue

# Если конфликт в src/ файле:
# Прочитать оба изменения, применить оба, не терять чужую работу
git rebase --continue

git push --force-with-lease origin issue-<N>-<slug>
```

## 6. Коммуникация через Issue

Если агент видит что его работа влияет на чужой открытый Issue:

```bash
gh issue comment <ЧУЖОЙ_ISSUE> \
  --body "ℹ️ Heads up: мой PR #<N> меняет <файл/функцию> которую ты используешь. 
  Merge планируется сегодня. Учти при rebase."
```

## 7. Пример параллельной работы (без конфликтов)

```
Агент A берёт #57 (zone:ci)     → трогает только .github/workflows/
Агент B берёт #42 (zone:gui)    → трогает только ui/*.slint
Агент C берёт #103 (zone:docs)  → трогает только README*.md

✅ Нет пересечений → merge без конфликтов
```

```
Агент A берёт #57 (zone:ci)
Агент B хочет #44 (zone:ci)     → СТОП: зона занята агентом A
Агент B берёт #90 (zone:tui)    → OK: другая зона

✅ B выбрал свободную зону
```

## 8. Чеклист перед взятием задачи (multi-agent)

```
[ ] Issue имеет label `agent`
[ ] assignee пустой
[ ] Зависимости (если есть) — все closed
[ ] Зона файлов — нет активного Issue в той же зоне
[ ] git pull origin main (актуальная база)
[ ] Создать ветку от свежего main
```

---

## 9. Человек-координатор

Человек (синтографист) управляет приоритетами через:
- Назначение labels `priority:high`
- Добавление/снятие label `blocked`
- Закрытие Issues которые стали неактуальными
- Слияние PR в main

Агент не принимает решения о приоритетах — только выполняет.
